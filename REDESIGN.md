## 0. v2 で何が変わったか（先に結論）

参考にした外部モデル（Amazonアソシエイト×AI自動運用のロードマップ）を Threads 用に翻訳した結果、
v1 の診断は正しかったが **処方箋が3点間違っていた**。

| 論点 | v1の結論 | v2の結論 | 変更理由 |
|---|---|---|---|
| 外部リンク | 週7本中1本だけ許可 | **毎回貼る。ただしルート投稿ではなくリプ（2投稿目以降）に置く** | リーチを絞るのはリンク付き"ルート投稿"。ツリーのリプに逃がせば、リーチを守ったまま毎回送客できる。週1に絞る必要がなかった |
| 投稿頻度と在庫 | 1日1本（記事在庫が足りない） | **1記事 → 3切り口 → 月/水/金の3日に分散** | 記事1本で3日分。週1記事ペースと消費が釣り合う。在庫問題が構造的に解ける |
| 切り口 | ANGLES 10種を回す | **3型（体験/比較/質問）に集約**。10種はサブ切り口に降格 | 10種だと1種あたりのサンプルが貯まらず、永久に答え合わせできない。3型なら3週で各9本 |
| 判断サイクル | 週次で判断 | **週次は記録だけ。判断は月次** | 1週間の数字はサンプル不足。週次で判断すると、ノイズを追って毎週方針が変わる |

そして v1 が見落としていた最大のもの:

> **7/19 の「案A/案B/案C 同日3連投」は、生成の失敗ではなく運用の失敗だった。**
> あれは「1記事を1週間かけて3回、違う切り口で告知する」ための3パターンであって、
> 同じ日に3つ出すために作られたものではない。仕組みは合っていて、配り方だけが間違っていた。

## 1. 目指す形

### 1-1. パイプライン

```
[日曜21:00] 週次生成 … 記事1本を選び、3型（体験/比較/質問）で3投稿ツリーを3セット生成
              ↓ 機械フィルタ（内部ラベル/定型/重複を自動リジェクト）
[日曜の夜]   週1回の承認 … Notionで3本まとめて○×（30秒×3）
              ↓
[月/水/金 08:00] 投稿 … 1セットずつ。直前に重複最終チェック
              ↓
[毎日15:00]  回収 … フォロワー数スナップショット + 48h経過分のinsights
              ↓
[月曜21:00]  週次 … 数字を記録するだけ。判断しない
              ↓
[月末]       月次30分 … 続けること1つ / やめること1つ / 新しく試すこと1つ を決める
```

### 1-2. ツリーの構造 ★ここが本丸

外部モデルの「訴求の3層（共感 → 使い勝手 → お得感）」を、そのままツリーの3投稿にマップする。

| | 役割 | 中身 | リンク |
|---|---|---|---|
| 1投稿目（ルート） | **共感** | 「これ、自分もずっと困ってた」。読者の悩みを言語化するだけ。ここで完結して読める | **絶対に貼らない** |
| 2投稿目 | **使い勝手** | 具体的にどうしたか。数字・固有名詞・実際のシーンを1つ以上 | **ここにブログURL** |
| 3投稿目 | **お得感 / 背中押し** | 「結局いくら得したか」「誰に向いてないか」 | なし |

why: ルート投稿だけが Threads のおすすめに乗る。ここにリンクがあると配信が絞られ、
かつ読者から見て「宣伝アカウント」に見える。リプに逃がせば、
**リーチを守ったまま、興味を持った人だけがリンクに到達する** = 送客の質も上がる。

### 1-3. 3型（1記事を3日に配る）

| 型 | 曜日 | 型の定義 |
|---|---|---|
| 体験型 | 月 | 自分の使用感・失敗談を1シーンだけ切り取る |
| 比較型 | 水 | 他の選択肢と何が違ったかを箇条書きで見せる |
| 質問型 | 金 | 「〜で困ってませんか？」と読者に投げる |

既存の ANGLES 10種は捨てず、**各型の中でフックの言い回しを変えるためのサブ切り口**として残す
（例: 体験型 × 失敗談 / 体験型 × 数字ショック）。分析の主軸はあくまで3型。

### 1-4. 決定事項

- 投稿頻度: **週3本（月/水/金 08:00 の1枠）**
- 生成: **週1回（日曜21:00）に3セットまとめて**
- 承認: **週1回まとめて。機械フィルタ通過後、人は面白さだけ○×**
- ハッシュタグ: **0〜1個**（Threads はトピックタグが実質1つしか効かない）
- 外部リンク: **毎回。ただし2投稿目のみ**
- ルート投稿にリンク・ハッシュタグ2個以上・記事タイトル丸写しがあったら**機械リジェクト**

## 2. 憲法（CLAUDE.md）をリポジトリに置く

いま NG表現も訴求ルールも `generate_tree.py` の SYSTEM_PROMPT に埋まっている。
生成以外の工程（週次レビュー、返信候補出し）には効かない。
外部モデルの「CLAUDE.md = チーム憲法」に倣い、リポジトリ直下に切り出す。

```
## 絶対NG表現
- 誇大な断定（必ず・確実に・誰でも・秒で・100%）
- 「今だけ」「限定」「損します」の煽り
- 「読まないと損」系の脅し文句
- 根拠のない数字・実績
- 医薬品・健康食品の効能を断定する表現
- 「」以外の記号での強調
- 「いかがでしたか」「〜について解説します」

## 訴求の優先順位
共感 > 使い勝手 > お得感（この順番を崩さない）

## 読者への姿勢
- 押し売りしない / 買う前に価値が伝わる構成 / 必ず「向いてない人」も書く

## 各工程の境界
- 生成は「投稿案」まで。公開はしない
- 最終承認は必ず人間（ゆうと）が行う
```

失敗するたびに1行ずつ追記していく前提で運用する。

## 3. 指標と判断サイクル

### 前提：Threads の役割は「記事への送客」であって、高単価商品への階段ではない

2026-07-21 決定。LINE公式は挟まない。note・ココナラへも繋がない。
chanko06.com の記事在庫は乾太くん・シャワーヘッド・エアコン・電力会社・マットレス等の
**実体験レビューと比較**で、収益は記事内のアフィリエイトで完結する。
読者は「その商品を買うか迷っている人」であって、note有料やFP講座の読者層とは重ならない。
無理に繋ぐと、記事の信頼を削って別ジャンルへ流すだけになる。

したがって外部モデルの4段動線のうち、当てはまるのは2段目まで:

```
1段目 記事で「役立った」体験を得る
2段目 Threadsで日常のこの人を見る（← ここが上限。3段目以降は作らない）
```

### 主指標（最終）
- **Threads経由のブログセッション数**（GA4の参照元で取得）
- その先の **アフィリエイト成約**（月次で見る）

### 先行指標（主指標の母数と原因）
- **週次フォロワー増加数**（毎日スナップショットして差分）… 送客の母数。減ったら主指標も必ず落ちる
- ルート投稿の views / likes / replies
- **3型別**の平均パフォーマンス ← 答え合わせの主軸
- サブ切り口別（10種）の平均 ← サンプルが貯まってから見る

why: フォロワーを主指標に置くと、伸びる投稿＝バズる投稿を追ってしまい、
記事と関係ない話題に流れる。逆に送客だけを主指標にすると v1 の「全投稿が宣伝」に戻る。
**フォロワーは母数、送客が成果**、という2階建てで見る。

### 判断の単位
- **週次（月21:00）: 記録だけ。方針を変えない**
- **月末30分: 数字を見て「続ける1つ / やめる1つ / 新しく試す1つ」を決める**
- 改善は3ヶ月単位で見る（1ヶ月目=気づく / 2ヶ月目=試す / 3ヶ月目=定着させる）

### やってはいけない判断
- 1週間の数字だけで決める（サンプル不足）
- 感覚だけで「この投稿は良い/悪い」を決める
- 全部を毎月手直しする（改善対象は1〜2点に絞る）

## 4. アカウントの土台（1回だけやる）

外部モデルの「プロフィールで信頼を担保する」をそのまま適用。投稿を直す前にここを直す。

### プロフィール4行
1. 今の生活の1コマ（小さくていい）
2. 1年前の自分の悩み（具体的な数字を入れる）
3. 何を試して何が刺さったか（失敗も入れる）
4. 読者と共有したい価値観を1行

### ピン留め投稿
「何を発信する人か」が3秒で分かる1本を固定する。

### 記事末尾のCTA（chanko06.com 側）
- 今日の記事の一番の学びを1行で言い直す
- 次にとる小さな1歩を提示する
- その1歩を助けるリンクを1つだけ置く（→ Threadsプロフィール）
