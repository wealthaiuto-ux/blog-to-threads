## 1. 診断 — 何が壊れていたのか

### 1-1. 表面的な症状
- 7/19 に `【案A・体験談型】` 等の内部ラベルが付いたまま3連投
- 投稿40件中、6本の記事がちょうど3回ずつ使われていた
- 反応データを1件も回収していない

### 1-2. 根本原因（3つ）

**① 「修正する対象」がどこにも定義されていなかった** ★これが本丸

仮に数字が取れていても、「質問型が伸びた」と分かった後に
**どのファイルのどの行を書き換えれば明日の投稿が変わるのか** が決まっていない。
だから数字を見ても何も変わらない。レポートが増えるだけになる。

同じ問題が @uto_ai_publisher の夜間ルーチンにも起きている。
毎晩「発見TOP3」「明日試す仮説TOP3」を出力しているが、
それを次の生成に反映する経路がなく、毎晩ゼロから考え直している。

**② 在庫の偏り（在庫不足ではない）**

公開記事は35本。うち **15本は一度もThreadsに使われていない**。
その一方で6本が3回ずつ使われていた。記事選択ロジックの欠陥であって、在庫は最初から余っていた。
さらに「1記事＝1投稿」でしか使っていなかった。**35本 × 3〜4型 ≒ 105〜140本**、毎日1本なら3〜4ヶ月分ある。

**③ 検証されないまま埋め込まれた前提が3つあった**

| 前提 | 実際 |
|---|---|
| ブログ記事がネタ元である | 因果が逆でもよい。ネタは「買ったもの・払った額」であって記事ではない |
| 投稿はツリーである | 41投稿すべてツリー。単発の選択肢がコードに存在しない。伸ばす段階では単発が有利な場面が多い |
| テーマは「ライフハック」である | 広すぎて、3投稿見ても何の人か分からない |

（URL移転は301でも順位が落ちるリスクがあり、得られるのは整理された感覚だけで収益は増えない）。
この線引きが効くのは「これから書くもの」に対して。

### 2-4. 育児クラスタの扱い

**テーマの柱にはしない。「失敗談型・向いてない型」の素材として使う。**

why: 育児グッズの検索需要は枯れない（記事としては資産）。
だが Threads は検索と別物で、**書き手が熱を持てないネタは読者にも伝わらない**。
2024年の記事は「生後3ヶ月〜1年」の話で、今の生活からは2年前。鮮度がない。
ただし「2年前に6本まとめ買いして5本無駄にした」という失敗談なら、姿勢の側で一貫する。

初期配分は月4〜6本。**3ヶ月後、数字で比率を調整する**（意見で決め打ちしない）。

## 3. 運用モデル — ゆうとさんがやることは3つの○×だけ

前提: 稼働は夜のみ・時間が取れない。**判断材料も修正案もすべて自動生成し、人は承認だけ。**

| いつ | 時間 | やること |
|---|---|---|
| 毎晩 | **30秒** | 明日の投稿案1本を ○ / ×。× なら理由を1つ選ぶ（つまらない / 自分の言葉じゃない / 事実が違う） |
| 毎日 | 10分 | 返信（候補は自動でリストアップ） |
| 月末 | **3分** | playbook の修正案が1つ出るので ○ / × |

レポートを読み込む作業は入れない。週次レポートは読まなくてよい。

## 4. ループ — ここが v3 の中心

### 4-1. 修正対象を1個のファイルに固定する

`data/playbook.json` が **システムの設定値の唯一の置き場**。

```json
{
  "types": {
    "実測値型":     { "weight": 5 },
    "向いてない型": { "weight": 4 },
    "失敗談型":     { "weight": 4 },
    "質問型":       { "weight": 3 },
    "逆張り型":     { "weight": 2 },
    "1コマ型":      { "weight": 2 }
  },
  "themes": { "家電": 5, "電力・お金": 4, "住まい": 3, "育児": 1 },
  "updated_at": "2026-07-22",
  "changed": "初期値"
}
```

- **生成**は毎晩この weight を読んで型とテーマを選ぶ
- **月次レビュー**はこの weight を書き換える

これでループが閉じる。数字 → weight書き換え → 翌月の投稿が変わる → また数字が出る。
「修正」の実体がファイルの数字1個になるので、やったかどうかが曖昧にならない。

### 4-2. 全体フロー

```
[毎晩21:00] 生成
    playbook.json の weight で型とテーマを選ぶ
    記事在庫35本から、直近30日に未使用のものを優先
    機械フィルタ（後述）を通す
         ↓
[毎晩30秒] 人が ○ / ×  ← 人の作業はここだけ。×の理由も記録する
         ↓
[翌朝08:00] 投稿（投稿直前に重複最終チェック）
         ↓
[毎日15:00] 回収
    フォロワー数を1個記録 ／ 48h経過した投稿の views・likes・replies を取得
         ↓
[月曜] 週次レポート … 数字を並べるだけ。判断しない。読まなくてよい
         ↓
[月末] 月次レビュー
    型別・テーマ別の平均を出す → weight変更案を1つだけ出す → ○ で playbook.json 更新
         ↓
    翌月の生成が変わる（ループが閉じる）
```

### 4-3. 意図的な設計判断

**週次では判断しない**
7投稿では差が出ない。週次は「記録が取れているか」の確認用。判断は月1回。

**月次で変えるのは1つだけ**
一度に3つ変えると何が効いたか分からなくなる。1ヶ月に1箇所。

**× の理由も指標として扱う**
「自分の言葉じゃない」が続く → 生成プロンプトの問題。
「つまらない」が続く → 型かテーマの問題。
数字では取れない情報を人間の却下理由が運ぶ。前のシステムはここが完全に死んでいた。

### 4-4. 機械フィルタ（人が見る前に自動リジェクト）

- 内部ラベル混入（`【案`, `【パターン`, `| ちゃんこ`）
- 記事タイトルの丸写し（先頭30字一致）
- **ルート投稿へのURL混入**
- ハッシュタグ2個以上
- 直近30日の重複記事

→ 最大3回リトライ、通らなければ `status=needs_fix` で保存（人には出さない）

## 5. 投稿の設計

### 5-1. 形式

- **単発を基本にする**（ツリー前提を捨てる）。ルート投稿だけが配信に乗り、ツリーは読者に負荷をかける
- ツリーにする場合も、**ルート投稿だけで完結して読める**こと
- **ルート投稿にURLを置かない**。リンクを貼るならリプ（2投稿目）に
- ハッシュタグは 0〜1個（Threads は実質1つしか機能しない）

### 5-2. 6つの型と実例

型を6つに絞ったのは答え合わせのため。毎日1本なら**3週間で各3本、6週間で各6本**たまる。
10種類だと1種あたりのサンプルが貯まらず、永久に比較できない。

**① 実測値型**
> 乾太くん、ガス代がいくら増えるか怖くて2年迷ってた。
> 導入3ヶ月の実費を出したら、月＋1,100円だった。
> 3人家族・都市ガス・週7〜8回まわしてこの数字。
> 迷ってた2年分の「乾かない部屋着」のストレスは、月1,100円で消えた。

**② 向いてない型**（売り込まないほど信用される）
> シャワーヘッド、正直こういう人には向いてない。
> ・元から水圧が強い家
> ・家族全員シャワー派で回転が速い
> うちは子どもがシャワー嫌いで、ミストモードだけのために買った。
> そこにだけ効いた。それ以外は正直ふつう。

**③ 失敗談型**（育児クラスタはここで使う）
> エアコン工事、見積14万で頼んだのに当日に追加料金の話が出た。
> 3階建てだと配管が長くなる、と。
> 先に言ってくれよと思ったけど、聞かなかった自分のミスでもある。
> これから頼む人は「配管延長は見積に入ってますか」だけ聞いておくといい。

**④ 質問型**（リプが増える＝配信が伸びる）
> エアコンって何年で買い替えるのが正解なんだろう。
> うちは10年目で1台壊れて、結局2台まとめて替えた。
> まだ動いてたほうを替えたのは、正直もったいなかった気もしてる。
> みんな何年で替えてます？

**⑤ 逆張り型**
> 電気代は固定費だと思ってた。
> 電力会社を替えただけで年18,000円下がった。
> やったのは申し込みフォームを1回埋めただけ。工事も立ち会いもなし。
> 「固定費」って言葉、思考停止のサインかもしれない。

**⑥ 1コマ型**（フォローの動機はここで生まれる）
> 梅雨、前は憂鬱だった。
> 今は乾太くんがあるから、洗濯物のことを1回も考えなくていい。
> 天気予報を見る回数が減った。
> 家電って、時間じゃなくて「頭の中の占有率」を減らしてくれるんだなと思う。

### 5-3. NG（憲法 `CLAUDE.md` に切り出す）

```
## 絶対NG表現
- 誇大な断定（必ず・確実に・誰でも・秒で・100%）
- 「今だけ」「限定」「損します」の煽り
- 「読まないと損」系の脅し文句
- 根拠のない数字・実績
- 「」以外の記号での強調
- 「いかがでしたか」「〜について解説します」

## 訴求の優先順位
共感 > 使い勝手 > お得感

## 読者への姿勢
押し売りしない / 買う前に価値が伝わる / 必ず「向いてない人」も書く

## 境界
生成は「投稿案」まで。公開はしない。最終承認は必ず人間（ゆうと）
```

生成以外の工程（月次レビュー、返信候補出し）に効かない。リポジトリ直下に切り出す。

## 6. 指標

### 6-1. 最初の3ヶ月は「送客」をKPIにしない

アフィリエイト収益 = 記事PV × クリック率 × 成約率 × 単価。
Threadsが触れるのはPVだけで、しかも**検索で来る人（買う直前）とThreadsで流れてくる人（暇つぶし）は成約率が一桁違う**。
Threadsから送客してアフィ収益を伸ばそうとすると期待値が低い。

収益の観点で Threads に価値があるのは次の3つ:
1. **テーマ検証の場** ← 本命。どのネタが刺さるか記事を書く前に分かる＝書く記事の打率が上がる
2. **指名検索とE-E-A-T** … 記事の滞在時間・再訪が上がる
3. **将来の母数** … フォロワーが増えれば送客量も増える（後段）

### 6-2. 見る数字

| 段階 | 指標 |
|---|---|
| 主指標（先行） | **フォロワー増加数**（毎日スナップショット。過去に遡れないので記録開始が早いほど得） |
| 副指標 | インプレッション / いいね / リプライ |
| 比較軸 | **型別**（6型）と**テーマ別**（4クラスタ）の平均 ← 月次レビューの材料 |
| 人間側 | ×の理由の分布 |
| 後段（3ヶ月後〜） | Threads経由のブログセッション（GA4参照元） → アフィ成約 |

### 6-3. 期待値の正直な設定

**最初の1ヶ月は何も分からない。** 30本＝型ごとに5本では偶然の範囲。
意味のある差が見えるのは3ヶ月目。1ヶ月目「気づく」／2ヶ月目「試す」／3ヶ月目「定着させる」。
**最初の2ヶ月は伸びが実感できなくても正常。** ここで止めるのが一番もったいない。
